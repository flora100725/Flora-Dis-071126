# Agentic OS 醫療器材審計平台與代理人作業系統技術規格書與使用者手冊

## 1. 系統願景與架構概述 (System Vision & Architecture Overview)

### 1.1 系統使命與核心價值 (Mission & Core Values)
隨著全球醫療器材供應鏈的日益複雜化以及高風險（Class III）植入性醫療器材（如心律調節器、人工關節等）在臨床應用中的廣泛普及，如何精準追蹤醫材生命週期、防範無申報販售、確保許可證效期合規，已成為各國食品藥物管理局（FDA）面臨的最嚴峻挑戰之一。

本**Agentic OS 醫療器材審計平台（Agentic OS & Medical Device Audit Platform）**，旨在建構一個企業級、高保真的主動式行政與法規審計終端。本系統首創將「代理人作業系統（Agentic OS）」的動態沙盒理念，與深度的多維度關聯供應鏈審計引擎相結合。透過大語言模型代理（LLM Agents）的主動推理、網頁檢索能力與代碼執行能力，本平台能夠化被動審查為主動監管，協助 FDA 官員、合規稽查員與醫療機構管理者，在數秒內從海量申報數據中抽絲剝繭，精確識別潛在的安全漏洞、過期產品、偽冒醫材或供應鏈瓶頸，從而捍衛公共衛生安全與患者生命財產尊嚴。

本平台的核心價值主張在於：
1. **法規審計確定性**：利用確定性的數據庫規則與多維交叉比對算法，即時標記過期許可證、未授權經銷商、進出貨數量不對等的「零申報」灰色地帶。
2. **代理人自主化自主定製（Agentic Customizability）**：允許非技術背景的監管官員使用自然語言編寫監管規則（即 `skill.md` 技能描述），並在 live 數據池中自動進行代碼生成、執行、比對與多模型並行驗證。
3. **頂級數據視覺化（WOW Vis）**：將極其複雜晦澀的進出貨與物流流向日誌，轉化為包含交互式地理資訊系統（GIS）地圖、多層級桑基能量分流圖（Sankey Flow）在內的六大高保真動態圖表，實現對醫材流向「上帝視角」式的掌控。

#### 1.2.2 前後端通訊與安全架構
* **API 密鑰隱蔽**：所有的 Gemini API 密鑰均嚴格託管於伺服器端的 `process.env.GEMINI_API_KEY`，前端絕不暴露任何 API Key。
* **數據流向洗滌（Data Sanitization）**：當 FDA 稽查員上傳異構進出貨申報表（非標準格式的 Excel/CSV）時，系統會先透過伺服器端的「AI 數據標準化精靈」進行預處理，對敏感的患者隱私數據（如病歷號、姓名）進行本地脫敏與遮蔽，僅將脫敏後的品名、批號、許可證號與數量結構傳遞至大模型進行 Schema Alignment。
* **即時雙向狀態機**：前端通過一個動態的狀態管理器維護 Agent 的即時微狀態（Micro-States），包括：`[Idle]`（空閒靜止）、`[Thinking]`（模型推理中，顯示 Token 生成速率）、`[Calling Web Search]`（網頁實時 Grounding 檢索）、`[Executing Code Sandbox]`（代碼沙盒運行中）以及 `[Streaming Output]`（流式文字渲染中），極大提升了行政用戶的操作信任感。

## 2. 核心功能模組技術詳解 (Core Functional Modules)

### 2.1 模組一：Agentic OS 智能代理與技能（Skill）運行時
本模組是整個平台的大腦，將傳統靜態的法規規則手冊轉化為可執行、可自適應優化的「智能技能（Skills）」。

#### 2.1.1 技能文件規格（`skill.md`）
系統將每一項監管任務定義為一個符合特定 Markdown Schema 的 `.md` 檔案。其標準結構如下：
```markdown
# 技能名稱：[例如：高風險醫材無申報販售深度稽核]
## 1. 監管背景與法律依據
- 依據《醫療器材管理法》第 25 條與第 62 條...
## 2. 稽查邏輯與約束條件
- [條件一] 檢查出貨數據中的許可證是否在《許可證主檔》中失效。
- [條件二] 若醫材為 Class III，必須強制校驗 14 位 UDI 序號（SN）。
- [條件三] 對比特定進貨商與出貨商在同一批號下的進出總量。
## 3. 輸入數據 Schema 要求
- 支援 PurchaseRecord 與 DistributionRecord 的聯集。
## 4. 預期 JSON 輸出結構
```

#### 2.1.2 代理人協同編輯與代碼生成
* **自然語言技能修訂（NLP Refinement）**：用戶可直接在左側的技能編輯器中，輸入文字指令（如：「請幫我修改這項技能，增加對『過期 30 天內產品可寬限退回』的特殊篩選機制」）。伺服器端將調用 `gemini-3.5-flash`，理解修改意圖並重新生成結構嚴謹的 `skill.md`。
* **Google ADK 工具深度整合**：
  * **內建代碼執行（Code Execution）**：當運行審計任務時，大模型會自動生成 Python Pandas 腳本，在伺服器安全的計算沙盒中運行。這解決了大語言模型在處理萬級行數據時「數學計算不精準、漏報多報」的硬傷。Python 沙盒會執行諸如：`df_purchase.groupby('Batch').sum() - df_dist.groupby('Batch').sum()` 等矩陣對比。
  * **谷歌搜索 Grounding（Google Search）**：大模型被授予實時網頁檢索權限。在審查特定許可證時，大模型會主動向 TFDA 官方網站、衛福部公告或美國 FDA MAUDE 數據庫發起實時檢索，校驗該型號在過去 48 小時內是否有未登記的召回事件（Recall Alerts）。

#### 2.1.3 多模型與多技能並行沙盒
為了確保審計結論的絕對客觀與可追溯性，平台設計了「對比矩陣控制台」：
* **雙模型對比（Model A/B Test）**：可選擇將同一份進出貨數據與技能，同時發送至 `gemini-3.5-flash` 與 `gemini-2.5-pro`。系統將並排顯示兩者的分析結果，並輸出精準的技術效能指標：**處理延遲（Latency Ms）**、**提示詞 Token 消耗（Input Tokens）**、**生成 Token 消耗（Output Tokens）** 及 **預估 API 運行成本（Cost）**。
* **雙技能對比（Skill A/B Test）**：使用同一模型，對比「常規合規稽查」與「極端欺詐與黑市流向稽查」兩項技能的洞察深度，方便 FDA 決策層評估最優監管策略。

### 2.2 模組二：醫材申報數據清洗與進階合規引擎
本模組負責將真實世界中混亂不堪、格式各異的廠商申報表轉化為高清潔度的結構化稽查矩陣。

#### 2.2.1 智能數據洗滌精靈（AI Harmonization Wizard）
當外部申報商上傳的 Excel 包含「供應商名稱」、「品名縮寫」等非標準欄位時，AI 數據洗滌精靈會引導用戶進行語意對齊：
* **語意對齊（Semantic Mapping）**：例如，系統自動識別 `Vendor_Name` 為「申報業者」，`Lic_ID` 為「許可證號」，並在介面上繪製出兩側欄位對接的動態矢量連線圖。
* **格式標準化（Format Normalization）**：自動將隨意輸入的日期（如 `2026/4/29`、`29-Apr-2026`、`民國115年4月29日`）統一轉化為符合資料庫規範的 `YYYYMMDD` 格式。
* **可視化確認與滑桿分頁**：數據清洗後，前端提供可調整的即時分頁預覽組件（支援滑桿調整：10、20、50、100、500 行或全量展示），並支持一鍵將清洗後的數據導出為帶有 UTF-8-SIG 的標準 CSV 或 JSON 文件。

#### 2.2.2 高階多維度聯控過濾面版
為了方便官員迅速鎖定目標，本面版內置多個與底層 Master 資料庫連動的動態過濾器：
* **許可證動態補全**：與《許可證主檔（License Master）》進行外鍵關聯，用戶鍵入關鍵字即可動態補全標準格式的許可證號（如：`衛部醫器輸字第030747號`）。
* **醫材風險級別快速篩選**：一鍵切換 Class II（中度風險）與 Class III（高度風險，如植入物、心臟導線）數據，當選擇 Class III 時，界面自動高亮未填寫「產品序號（SN）」的違規記錄。
* **醫事機構與經銷商外鍵聯鎖**：自動過濾無效或未在衛福部登記的醫事機構 ID（如台北榮總 `A00002`、台大醫院 `A00013`）。

## 3. 六大高保真交互式視覺化圖表設計 (Visualizations)

為確保「上帝視角」的監管體驗，平台 Bento Grid 佈局中嵌入了六個完全基於真實數據渲染的交互式 SVG 圖表。

### 3.1 圖表一：進出貨多層級桑基能量分流圖（Sankey Flow Chart）
* **核心監管目標**：揭示醫療器材從原始製造商、進口經銷商、二級代理商到最終臨床醫院的完整權限流轉路徑。
* **交互設計**：
  * 支持節點懸停。滑鼠移至「美敦力台灣分公司」節點，與其相連的所有上游「進口批次」與下游「受領醫院」的流向管道將自動變為高亮。
  * 異常流動標記：若管道流向了「未登記非合規經銷商」，該管道將呈現閃爍的警示色（如玫瑰紅），並彈出浮動小視窗展示違規交易編號。

### 3.2 圖表二：GIS 醫事機構地理資訊分佈地圖（Gis Map）
* **核心監管目標**：將全國物流流向在地圖上進行精確空間定位，幫助官員排查區域性醫材異常聚集或黑市轉運點。
* **技術實現與交互**：
  * 地圖背景使用精緻的高對比暗色調，繪製出台灣各縣市的精細行政邊界線。
  * **動態信標（Beacons）**：使用綠色圓點標記「合規醫事機構（如台大醫院）」，使用藍色圓點標記「經銷商倉庫」。
  * **動態物流射線**：當在下方的進出貨表中選擇某筆交易時，地圖上會即時繪製出一條從供應商指向收貨醫院的動態粒子射線，並在懸停時顯示兩者間的物理距離與運輸所需溫度、時間等中繼參數。

### 3.3 圖表三：進貨 vs. 出貨月度體積雙軸對比圖（Dual-Axis Volumetrics）
* **核心監管目標**：一眼看穿「囤貨惡意炒作」與「無申報走私進口」的黑天鵝事件。
* **技術實現**：
  * 左側 Y 軸表示「進貨申報總量（Bar，柱狀圖）」，右側 Y 軸表示「出貨分銷總量（Line，折線圖）」。
  * **偏離度陰影（Delta Shadow）**：當折線（出貨）顯著高於柱狀（進貨）時，兩者之間的夾角會自動填充為半透明的警示紅色陰影，代表「市場上正在流動未經進口申報的黑市醫材或仿冒品」；反之若進貨遠大於出貨，則高亮黃色陰影，代表「經銷商可能正在惡意囤積防汛、防疫物資」。

### 3.4 圖表四：醫材批號保存期限到期熱力老化圖（Expiry Aging Heatmap）
* **核心監管目標**：預防過期無菌醫材被植入人體造成嚴重醫療事故。
* **技術實現**：
  * 採用 bento 矩陣佈局，按醫材子類別（如 E.3610 心臟節律器、H.3510 人工關節）平鋪色塊。
  * **色階過渡**：綠色代表安全期（保存期限大於 2 年），黃色代表中度預警（保存期限 180 天至 2 年），橙色代表高度預警（保存期限 30 至 180 天），紅色且伴隨呼吸閃爍效果則代表已過期。
  * 點擊任何子類別色塊，系統會彈出該類別下所有即將到期的具體批號、目前存放倉庫與受領醫院。

### 3.5 圖表五：法規異常分類與風險權重條形圖（Risk & Anomaly Bar Chart）
* **核心監管目標**：為 FDA 官員每日的執法優先級提供量化指標。
* **技術實現**：
  * 橫軸為異常類型（零申報、過期許可證、Class III 缺序號、非法渠道、跨國召回未處置），縱軸為加權後的「綜合危害指數（Risk Score）」。
  * 支持點擊下鑽（Drill-Down）。點擊「Class III 缺序號」長條，主數據表會自動進行過濾，僅顯示涉案的高風險產品，實現秒級排查。

### 3.6 圖表六：經銷商與醫療機構交易網絡矩陣圖（Transaction Network Matrix）
* **核心監管目標**：揭示是否存在利益輸送、非法利益綑綁或異常集中的採購行為。
* **技術實現**：
  * 採用熱力矩陣形式，縱軸為所有註冊經銷商，橫軸為所有醫院機構，交叉點的顏色深淺代表交易頻次與金額。
  * 點擊高亮色塊，可立刻還原該網絡下的所有交易流水，提供強大的反洗錢與反黑市鏈條穿透审计能力。

## 4. 三個 Wow 級 AI 創新功能設計 (Additional Wow AI Features)

為了使平台在監管科技（RegTech）領域達到全球領先的頂尖水準，我們在現有架構基礎上，設計了以下三個具備顛覆性體驗的 WOW 級 AI 核心功能。

### 4.1 WOW 功能一：醫療器材影像智慧 UDI 辨識與包裝完整度稽核 (Medical Device Image Smart UDI Recognition & Packaging Integrity Audit)
* **功能概述**：
  在實際稽查中，FDA 基層執法人員常需前往海關倉庫或醫院手術室進行實物抽檢。傳統方式是人工肉眼對照包裝盒上的 UDI 條碼與系統數據，費時且極易出錯。本功能利用多模態大模型（如 `gemini-1.5-pro` 或 `gemini-2.5-pro`），賦予平台實物圖像審計能力。
* **技術實現與工作流**：
  1. **實拍影像接入**：執法人員使用手機或平板，直接在平台「實物稽核」入口拍攝或上傳醫材外包裝的照片（支持標籤、一維條碼、二維 UDI 碼）。
  2. **多模態提取與 OCR 解析**：伺服器調用多模態 Gemini，不僅提取一維/二維 UDI 條碼，還會對包裝上的微小文字（如英文名、批號 `LOT`、失效日期 `EXP`、滅菌方式）進行高精確度的 OCR 解析與語意提取。
  3. **實時合規交叉校驗**：系統自動拿提取到的實物數據，與「進貨申報表」和「許可證主檔」進行三方比對。
  4. **外包裝完整度與假冒評估**：AI 還會評估照片中包裝盒的物理狀態。若檢測到包裝邊角破損、封條有二次粘貼痕跡、或是標籤排版與官方註冊標準樣張有細微偏差（如字體不對、商標位置偏離），AI 會發出視覺警告：「*標籤排版與 TFDA 備案樣張不符，假冒風險度 87%，建議扣留扣押並送檢。*」

### 4.2 WOW 功能二：醫療器材仿單語意合規性對比與 Off-Label 預警引擎 (Medical Device IFU Semantic Compliance & Off-Label Marketing Detection)
* **功能概述**：
  進口醫材的英文「使用說明書與仿單（Instructions For Use, IFU）」通常多達數百頁，包含極其複雜的適應症、禁忌症及操作步驟。部分不肖廠商在翻譯中文仿單或在台進行推廣時，可能會刻意誇大療效、隱瞞部分嚴重禁忌症，或進行未經核准的適應症推廣（Off-label marketing）。本功能利用大語言模型的長文本（Long Context）理解與精準翻譯對比能力，自動進行中英文仿單的語意差異安全稽查。
* **技術實現與工作流**：
  1. **雙邊文檔導入**：系統允許 FDA 審查官員同時拖入兩份 PDF 文件：A 檔為「國外製造商原始英文 IFU」，B 檔為「申請在台上市的中文核准仿單或宣傳手冊草案」。
  2. **百萬 Token 語意比對**：利用 Gemini 的百萬級上下文窗口，將兩份文檔進行精準段落對齊與多層次語意語素比對。
  3. **自動亮點標記與預警（Smart Diff）**：
     * **禁忌症隱瞞警告**：若英文原版寫道 `Not recommended for patients under 12 years old`，但中文版將其翻譯為「兒童使用時需謹慎」，AI 將高亮標記並提示：「*嚴重語意降級：原廠限制 12 歲以下禁用，中文仿單刻意淡化為謹慎使用，涉嫌擴大適用人群。*」
     * **適應症越界（Off-Label）預警**：若宣傳手冊中提及的臨床療效超越了 TFDA 許可證登記的核准範圍，AI 會自動列出法規條文進行警示，極大提高了藥政審查官員的審件速度與精準度。

### 4.3 WOW 功能三：基於 Multi-Agent 聯動的跨國非法醫材流向主動偵測與社群輿情預警 (Multi-Agent Proactive Underground Flow & Social Vigilance Pre-warning)
* **功能概述**：
  許多非法醫材（如未經核准的醫美玻尿酸針劑、仿冒的家用血糖測試片、走私的隱形眼鏡）並不會主動向 FDA 的正式系統申報，而是隱密地流竄於各大非官方電商平台、社群媒體、醫美論壇甚至暗網。本功能首創「主動防禦」理念，在伺服器端部署一組主動巡邏的 Multi-Agent（多智能體系統），將監管觸角延伸至正規渠道之外。
* **技術實現與工作流**：
  1. **任務分工（Agent Role-Play）**：
     * **Scout Agent（偵察代理）**：利用 `google_search` 工具與自定義爬蟲，每日定時在 PTT、Dcard、臉書社團、蝦皮購物等平台，檢索「醫美針劑、低價血糖儀、原裝平行輸入」等關鍵字。
     * **Audit Agent（審計代理）**：將 Scout Agent 收集到的賣家貼文、產品照片、價格等，與「TFDA 許可證 Master 庫」和「合法藥商登記冊」比對。
     * **Profiler Agent（畫像代理）**：對異常線索進行聚類，勾勒出潛在的地下走私鏈條。
  2. **地下流向預警看板（Underground Flow Map）**：當 Agent 偵測到某未註冊賬號（如：*Line id: beauty_secret*）正在大量兜售某型號 Class III 玻尿酸，且價格僅為市場均價的 30% 時，系統會在 GIS 地圖上以「橙色骷髏信標」標記，並推播給 FDA 稽查官員：「*發現非法銷售網絡：涉案人員正在北部地區醫美診所私下供貨。建議啟動線下突擊執法。*」

為了保障平台在極端或錯誤輸入下依然具有 100% 的高可用性，研發維護團隊必須執行以下 QA 防禦規範：
1. **異步 SDK 延遲初始化（Lazy SDK Initialization）**：
   伺服器在啟動時，不得因缺少 `process.env.GEMINI_API_KEY` 而導致 Node 進程崩潰。所有的 `@google/genai` 實例必須採取 Lazy 初始化模式，即在特定的 API 路由被請求時，才進行 Key 的存在性檢查。若 Key 缺失，向前端返回清晰的 412 錯誤碼及對應的配置引導，前端在卡片內部優雅展示：「*AI 功能暫未配置 Secrets Key，請在系統設置中導入，本地過濾審計功能仍可照常使用。*」
2. **多重 React 錯誤邊界（Nested Error Boundaries）**：
   將 **Sankey Flow**、**Gis Map**、與 **AnalyticsCharts** 三大圖表組件分別包裹在獨立的 `ErrorBoundary` 中。如果用戶導入了極端畸形的 CSV 數據，導致 Recharts 內部發生渲染異常，錯誤僅會被限制在該圖表卡片內部，卡片會優雅降級為灰色佔位符並提供「一鍵重置為標準 Demo 數據」的按鈕，而不會導致整個 Agentic OS 桌面發生雪崩式白屏。
3. **可視化回退保護（Visual Fallback Protection）**：
   對於地圖或圖表，若緯度（Latitude）或經度（Longitude）數據解析為 `NaN`，系統必須提供默認的中繼站地理坐標（如衛福部坐標：`25.0521, 121.6163`），並將該點標記為「未知坐標站點」，絕對禁止直接拋出 React 崩潰錯誤。
