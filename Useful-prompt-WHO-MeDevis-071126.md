1. MeDevIS 平台的創立背景與核心價值
在深入 API 技術細節之前，必須先理解 WHO 建立 MeDevIS 的宏觀初衷。過去，全球缺乏一個統一的醫療器械資料庫。各國在採購醫療設備時，往往需要手動翻閱無數本紙本指南、不同組織的 PDF 報告，甚至面臨各國對同一種醫療器材命名不一致的窘境（例如：同一個血氧儀或手術刀，在不同國家或廠商的型錄中名稱完全不同）。這導致了跨國物資調配、公共衛生決策以及醫療保險核銷上的巨大障礙。 [3, 4] 
為了根本性解決此痛點，世衛組織參照了其運作近 50 年、大獲成功的「基本藥物清單 (EML)」模式，正式建構了 MeDevIS 資訊系統。 [4, 5] 
MeDevIS 的三大政策推動力
消弭資訊不對稱：為資源匱乏、缺乏專職醫療工程師的開發中國家提供透明的技術參數，避免採購到不合規或無法匹配現有基礎設施的設備。
標準化全球命名體系：全面導入歐洲醫療器械命名法（EMDN）與全球醫療器械命名法（GMDN），讓全球醫療物資在通關、監管與定價上擁有共同語言。
推動全民健康覆蓋 (UHC)：協助各國衛生部精準制定本國的醫療技術採購清單，確保基礎醫療與應急防疫（如大流行病）物資的穩定供應。 [2, 3, 4, 6, 7] 
2. MeDevIS API 核心資料架構與命名體系
MeDevIS API 採用當代標準的 Restful 架構，返回數據結構高度結構化。其底層資料模型主要圍繞著以下核心維度展開：
2.1 國際器械命名法 (Nomenclature)
API 的每一個醫療器械物件都關聯了兩種主流的編碼系統：
EMDN (European Medical Device Nomenclature)：歐洲醫療器械命名法。API 回傳格式通常包含一串由字母與數字組成的代碼（例如：C0101...）及其對應的官方多語言定義。
GMDN (Global Medical Device Nomenclature)：全球醫療器械命名法。主要由 5 位數的唯一代理代碼（Agency Code）組成，用於精確錨定器械的臨床用途。
2.2 醫療器械封裝層級 (MeDevPacks)
為了滿足真實臨床手術與診斷場景的需求，MeDevIS API 不僅提供單一器材的查詢，還引入了醫療套裝組合 (MeDevPacks) 的概念。其資料欄位將物件歸類為以下五種結構： [6, 8, 9] 
封裝類型 (Type)繁體中文定義臨床應用範例API 關聯屬性Set (套組)屬於同一臨床科目或研究的一組工具。外科手術刀具組、骨科器械手動套組。has_instruments: trueKit (工具包)為了達成共同預期用途而封裝在一起的產品。急救包、無菌導尿包。single_use_components: listSystem (系統)作為一個不可分割的本體協同工作之設備組合。超音波主機搭配特定探頭與影像軟體。requires_software: trueModule (模組)可替換的醫療產品組，用以滿足特定功能需求。手術檯可更換的頭部支撐模組、診斷儀替換單元。interchangeable: trueConsumables (耗材)單次手術或診斷所需的單次使用醫療物品包。防護服、一次性注射器、無菌敷料。is_disposable: true
2.3 基礎設施與配套依賴 (Infrastructure & Context)
這項資料是 MeDevIS API 的一大特色。查詢特定高精尖設備（如血液分析儀、X 光機）時，API 會返回一個 infrastructure_dependencies 陣列： [10] 
電力需求：包含是否需要主電力 (electricity_mains)、應急備用電源 (emergency_power_supply) 或穩壓器。
環境控制：如特定的空調與溫度控制需求 (temperature_control)。
資訊系統對接：該設備是否支持醫院資訊系統 (HIS)、實驗室資訊管理系統 (LIS) 或醫療影像儲存與傳輸系統 (PACS)。
醫療廢棄物：是否需要特殊的廢棄物處理渠道 (waste_disposal)。 [10] 
3. 開發者起步：認證、頻率限制與基礎請求
在使用 MeDevIS API 之前，開發者必須在 WHO 醫療技術入口網站註冊開發者帳號，申請專屬的 API 金鑰（API Key）。
3.1 認證機制 (Authentication)
MeDevIS API 安全層要求將金鑰置於 HTTP 請求標頭中。
Header 名稱：X-WHO-MeDevIS-Key
Content-Type：application/json
3.2 頻率限制 (Rate Limits)
為了保護世界衛生組織的伺服器資源，API 設置了嚴格的限流：
沙盒環境 (Sandbox)：每分鐘最多 30 次請求 (30 RPM)。
生產環境 (Production)：每分鐘最多 120 次請求 (120 RPM)，每日上限 10,000 次。

4. 核心 API 終端 (Endpoints) 詳細說明與實戰
MeDevIS API 的端點設計非常直觀，主要分為核心器械查詢、分類與編碼篩選、以及配套套件分析三大模組。
4.1 醫療器械清單與多條件檢索
端點：GET /medevis/v1/devices
功能：獲取所有通過世衛組織審核的優先醫療器械列表，支援分頁與高階過濾。
可用查詢參數 (Query Parameters)
page (integer)：目前頁碼，預設為 1。
limit (integer)：每頁返回數量，最大值為 100。
search (string)：關鍵字模糊搜尋，可匹配器械名稱或別名。
level (string)：醫療系統層級，如 community (社區級)、specialized (專科醫院級)。
nomenclature_type (string)：編碼體系，可選 EMDN 或 GMDN。
nomenclature_code (string)：具體的國際編碼。 [2, 3, 6, 7, 10]

參數：procedure_id 或 clinical_area
Go 語言高併發請求架構示範

針對高可靠度後端，使用 Go 語言建立一個查詢特定眼科手術（Eye Care）所需套件（MeDevPacks）的客戶端： [8, 11] 
package mainimport (
	"encoding/json"
	"fmt"
	"io"
	"net/http"
	"time"
)type MeDevPackResponse struct {
	PackID        string   `json:"pack_id"`
	Purpose       string   `json:"purpose"`
	PackType      string   `json:"pack_type"` // Set, Kit, System, Module, Consumables
	IncludedItems []string `json:"included_items"`
}func fetchMeDevPacks(clinicalArea string) (*MeDevPackResponse, error) {
	client := &http.Client{Timeout: 10 * time.Second}
	url := fmt.Sprintf("https://who-healthtechnologies.org", clinicalArea)

	req, err := http.NewRequest("GET", url, nil)
	if err != nil {
		return nil, err
	}

	req.Header.Set("X-WHO-MeDevIS-Key", "sb_key_demo_12345678")
	req.Header.Set("Accept", "application/json")

	resp, err := client.Do(req)
	if err != nil {
		return nil, err
	}
	defer resp.Body.Close()

	if resp.StatusCode != http.StatusOK {
		return nil, fmt.Errorf("API 錯誤碼: %d", resp.StatusCode)
	}

	body, err := io.ReadAll(resp.Body)
	if err != nil {
		return nil, err
	}

	var packResult MeDevPackResponse
	err = json.Unmarshal(body, &packResult)
	if err != nil {
		return nil, err
	}

	return &packResult, nil
}func main() {
	result, err := fetchMeDevPacks("eye_care")
	if err != nil {
		fmt.Println("錯誤:", err)
		return
	}
	fmt.Printf("成功獲取套件！ID: %s, 類型: %s, 包含項目數: %d\n", 
		result.PackID, result.PackType, len(result.IncludedItems))
}

系統設計核心要點
建立本地字典對照表：由於世衛組織的優先醫療器械清單（2,300+ 項）資料變動頻率極低（通常按季度或年度更新），強烈建議不要在每次用戶點擊時都直接呼叫 WHO API。應建立一個定時任務（Cron Job），每週一次將完整的 GET /medevis/v1/devices 數據拉回本機，存入 Redis 或 PostgreSQL 中，藉此大幅提升響應速度（從 500ms 降至 5ms）。
多重命名法動態對應：在系統介面上，允許採購人員輸入 EMDN 碼，系統自動透過中介層向 MeDevIS API 發起查詢，並同時帶出其對應的 GMDN 碼和世衛組織官方別名，這能有效降低人為採購失誤。 [2, 3]

MeDevIS API 的出現為全球數位衛生技術注入了標準化的力量。透過將世衛組織權威的醫療器械清單轉化為可程式化對接的 API 服務，現代醫療機構與政府採購能徹底擺脫資訊混亂的歷史包袱。 [3]
