# 法規導向醫材來源流向 AI 追溯平台：系統技術規格書 (Technical Specification)

本規格書旨在針對台灣食品藥物管理署（TFDA）頒布之《醫療器材來源流向資料建立及管理辦法》，詳細規劃一款基於 Python、Streamlit 與 AI Agent 技術的**全功能本地端 Web App 稽查追溯平台**。本系統專為處理萬級以上規模（每年 10k+ 筆紀錄）之異質醫材流通大數據所設計，能依據法規勾稽「進貨（購買）」與「出貨（流向）」兩端申報網絡，並無縫整合「應申報醫材許可證主檔」與「特約醫療院所名冊」作為執法比對基準（Baseline）。

系統採用極致的架構設計與視覺 polish，透過互動式 UI 允許使用者動態載入本地檔案（CSV/JSON）、自訂時間區間與過濾準則，即時產出符合高階稽查核心需求的**三大情資模組、5 大 Wow 動態圖表、以及 5 大 AI 偵探功能**。

---

## 1. 系統願景與法規合規目標

### 1.1 法規背景背景與稽查痛點

根據《醫療器材來源流向資料建立及管理辦法》，特定高風險醫療器材（如第三等級植入式心臟節律器）之進口商、販賣業者（經銷商）及醫療院所（醫院/診所），必須定期且精準地申報醫材的「來源」與「流向」。然而，在實際執法與稽查中，TFDA 面臨以下重大痛點：

* **申報時間差與斷鏈（Discrepancy & Gap）**：經銷商已申報出貨，但終端醫院卻遲遲未申報進貨，或兩端申報之數量、批號（Lot No）、序號（Serial No）不一致。
* **萬級資料人工勾稽困難**：每年超過萬筆、甚至累計達百萬筆之異質結構資料，欄位夾雜雜訊（如引號、中文備註），傳統 Excel 無法進行精準的時序追溯與拓撲網絡分析。
* **缺乏預警與主動偵測機制**：無法即時找出未依規定申報之許可證持有者，亦無法在爆發重大瑕疵醫材事件時，於 3 秒內啟動批號級全台召回模擬（Recall Simulation）。

### 1.2 平台核心定位

本平台定位為「本地端高階執法與數據偵探 Web App」。完全基於 Python 生態系與 Streamlit 框架建構，確保數據完全封閉於本地端運作，保障高度敏感之醫療機構交易與法規資料安全。平台將透過高度自動化的 AI Agent，將雜亂的原始申報資料轉譯為具備執法法律效力之合規證據鏈。

### 2.1 資料導入層與彈性欄位對齊

平台提供直覺的拖拉式（Drag-and-Drop）檔案導入介面，支援以下四種本地資料集：

1. **進貨/購買明細 (Purchase Dataset - CSV)**
2. **出貨/流向明細 (Distribution Dataset - CSV)**
3. **應申報醫材許可證主檔 (Licenses Master - JSON/CSV)**
4. **特約醫療院所與販賣業者名冊 (Entities & Geolocation Master - JSON/CSV)**

### 2.2 大數據高速清洗管道 (Preprocessing Pipeline)

系統後端使用高效能向量化運算，導入資料時自動觸發以下清洗任務：

* **雜訊與特殊字元剔除**：使用正規表示式（Regex）自動將 `“美敦力”` 清洗為 `美敦力`；將型號內欄位如 `# 美敦力亞士爾磁振造影植入式心臟節律器-單腔 W2SR01` 精準切分為純淨型號 `W2SR01` 與品名備註。
* **時間格式強制標準化**：將整數或異質字元如 `20260429`、`2026/05/06` 統一轉換為 ISO 8601 日期格式 `YYYY-MM-DD`。
* **唯一法規流通交易雜湊鍵 (Transaction Hash Key)**：
系統會自動組合關鍵法規追溯欄位，並透過加密演算法生成全系統唯一的雜湊 ID：

$$\text{Transaction\_ID} = \text{MD5}(\text{申報業者} + \text{日期} + \text{許可證號} + \text{產品批號} + \text{產品序號} + \text{產品型號})$$



此 ID 將作為跨檔案、跨年度追蹤、AI 探針檢索之唯一索引（Index）。

## 3. 三大核心稽查情資模組設計 (Three Core Audit Modules)

### 3.1 模組一：申報統計分析模組 (特定時間區間分佈)

此模組專注於量化全台各申報主體之法規遵行頻率。使用者可自由拉動時間軸（例如：自訂 30 天、90 天、180 天或特定滾動區間）。

* **時間滑塊主動聚合 (Temporal Aggregation)**：系統根據使用者設定的時窗，動態將進貨與出貨資料於記憶體內進行時序滾動加總（Rolling Sum）。
* **申報主體覆蓋率計算**：
* **許可證持有者端**：比對「醫材許可證主檔」，統計出在該時間區間內，哪些公司（許可證所有人）有進口事實但「申報次數為零」或「數量發生斷崖式停滯」。
* **經銷商端**：統計各級販賣業者（盤商）的進出貨頻率，產出流轉率指標。
* **醫事機構端**：根據「特約醫療院所名冊」，動態統計全台醫事機構之進貨申報覆蓋率，並依據層級（醫學中心、區域醫院、地區醫院、診所）分組呈現。



### 3.2 模組二：申報異常分析與預警模組 (法規紅旗與斷鏈偵測)

此模組為本平台的「數字執法核心」，旨在透過強大的多表關聯邏輯與不一致性演算法（Discrepancy Algorithm），精準揭露違規事實：

* **單邊申報與幽靈節點偵測 (Single-Sided Declaration)**：
* **有出無進（盤商有報、醫院漏報）**：分析出貨明細中，經銷商報稱已將批號 $X$ 之心臟節律器交付予某醫院，但該醫院之進貨明細中在規定天數內完全無此筆紀錄。
* **有進無出（來源不明、幽靈通路）**：醫院報稱向代碼為 `C00306` 的廠商購入醫材，但名冊與出貨檔中完全查無該廠商或該廠商完全未申報出貨。此功能可直接鎖定「地下經銷商」或「黑市洗貨」風險。


* **申報時序落後（Latency Risk）**：自動計算 `建立日期` 減去 `收貨/交貨日期` 之天數差。若法規要求 15 天內申報，系統將自動將大於 15 天之紀錄標記為「超期違規」，並統計各業者的平均延遲天數（Average Latency Rate）。
* **數量與規格不對齊警示**：針對同一產品批號/序號，若進貨端申報數量與出貨端申報數量不對等（例如：盤商報出貨 10 組，醫院報收貨 8 組，且無退貨申報），自動觸發「數量不一致紅旗警示」。

### 3.3 模組三：特定產品供應鏈分析模組 (包含供應量與終端流向)

此模組針對「高風險特定醫材許可證」或「特定產品型號/次類別（如 E.3610）」進行深度的全台流通軌跡穿透。

* **批號追溯鏈結（Lot-Level Lineage）**：使用者輸入或篩選特定型號（如 `W3DR01`），系統自動串聯其從進口申報、經銷商層層轉手、一直到終端醫療院所的「全生命週期流向鏈」。
* **全台供應量與存量動態估算**：
根據進貨量、出貨量與退貨資訊，即時動態計算出每一時間節點上，特定醫材在全台各個經銷商與醫院倉庫內的「虛擬留存餘額」（Virtual Stock Balance）：

$$\text{目前庫存餘額} = \sum(\text{進貨數量}) - \sum(\text{流出數量}) + \sum(\text{退貨數量})$$



這能協助 TFDA 評估全台特定植入式醫材的實體分佈，防止發生不良反應事件時無法精準定位。

## 4. 5 大 WOW 視覺化圖表設計 (Advanced Visual Analytics)

本平台的視覺化圖表不採用單調的圓餅圖或長條圖，而是採用具備**高度互動、多維度下鑽（Drill-Down）與高對比執法美學**的動態圖表。所有圖表皆與 Streamlit 的前端篩選器連動。

### 4.1 Wow 圖表 1：全台醫材流向動態軌跡地圖 (Dynamic Supply Chain Flow Map)

* **視覺呈現**：結合「特約醫療院所與販賣業者名冊」之經緯度資料。當使用者篩選特定醫材許可證與時間後，地圖上會繪製出發自經銷商節點（藍點）指向各醫療院所節點（紅點）的**動態流向弧線（Sankey Paths on Map）**。
* **互動 polish**：線條粗細代表該區間的流通總數量。滑鼠懸停（Hover）在線上會即時跳出資訊卡，顯示「美敦力(B00047) $\rightarrow$ 台大醫院(A00013) | 數量: 12個 | 型號: W3DR01 | 批號: RNJ146480G2001」。

### 4.2 Wow 圖表 2：來源與流向申報斷鏈網絡圖 (Declaration Disconnect Network)

* **視覺呈現**：採用**拓撲關係網絡圖（Network Graph）**。將申報業者、供應商、醫院抽象化為圖形節點（Nodes），其間的進出貨關係為邊（Edges）。
* **執法 polish**：若兩端申報完全合規對齊，連線顯示為平滑的綠色；若發生「單邊申報（盤商有報、醫院未報）」或「數量不對等」，該網絡連線會斷開或呈現**高亮粗紅色閃爍**，並將涉嫌漏報的醫院節點放大，讓稽查員一眼看出全台執法漏洞之核心核心結點。

### 4.3 Wow 圖表 3：產品生命週期與過期風險熱圖 (Product Lifespan Expiry Heatmap)

* **視覺呈現**：依據製造日期、有效期間與保存期限，系統為各醫院與經銷商內現存的醫材設計一組**二維矩陣熱圖（Matrix Heatmap）**。縱軸為產品型號，橫軸為保存期限倒數區間（如：已過期、<30天、30-90天、>180天）。
* **互動 polish**：顏色由深紅（代表已過期且滯留於醫院未消耗/未回收）漸變至深綠（安全效期）。稽查員點擊深紅方塊，下方表格會即時下鑽（Drill-down），連動顯示出目前有哪些批號的節律器在哪些特約醫院有過期高風險。

### 4.4 Wow 圖表 4：申報時序延遲與漏報趨勢複合圖 (Reporting Latency Composite Trend)

* **視覺呈現**：左軸為折線圖，呈現特定區間內每日全台平均申報延遲天數；右軸為堆疊長條圖，呈現每日「已屆法規申報期限卻仍處於單邊斷鏈狀態」的潛在漏報數量。
* **互動 polish**：支援時間維度自由切換（日/週/月/季）。透過刷選（Brushing）功能，使用者在趨勢圖上圈選某個異常延遲高峰期，平台其他頁面的廠商列表會自動過濾為該段時間內表現最差的違規黑名單。

### 4.5 Wow 圖表 5：高風險植入物全台終端留存漏斗桑基圖 (Implants Retention Sankey)

* **視覺呈現**：高階**桑基圖（Sankey Diagram）**。左側流出端為全台總進口量（100%），分流至各個一級、二級盤商，再分流至各醫療院所，最右側終端則分為兩大流向：公立醫院留存、私立醫院留存、以及一個顯眼的灰色區塊——「流向不明/申報黑洞」。
* **互動 polish**：稽查員點擊「申報黑洞」分流線，系統會彈出泡泡圖，按比例放大顯示造成此申報黑洞的主要責任經銷商與漏報醫院排名。

## 5. 5 大 WOW AI 核心魔法功能設計 (AI Magics & Agent Features)

本平台的核心靈魂在於整合大語言模型（LLM）與 Agent 框架，讓系統不只是個死板的工具，而是一個高智商的「法規稽查數據偵探」。

### 5.1 AI Magic 1：法規合規自動審查與紅旗預警 (Automated Regulatory Compliance Red-Flagging)

* **功能機制**：AI Agent 於背景全自動執行 Pandas / Polars 代碼，掃描當前篩選區間內的萬筆資料，檢視是否符合《醫療器材來源流向資料建立及管理辦法》的所有條款。
* **Wow 產出**：自動生成結構化的「法規紅旗預警儀表板」，主動列出諸如：「*偵測到 3 家幽靈供應商編號*」、「*違反本辦法第 4 條之超期申報業者共 14 家*」等具體法律合規結論，並為每條結論標記法規條文依據。

### 5.2 AI Magic 2：全台醫材黑市/囤積/異常流向預測 (Anomaly Flow & Volume Detection)

* **功能機制**：結合統計學（Z-Score）與 AI 語意解讀。偵測特定批號醫材在極短時間內於複數經銷商之間進行「循環對調」、「反覆退貨再倒貨」，或某家小型診所突然購入極不合理之大量高風險第三等級醫材。
* **Wow 產出**：AI 會跳出高能警示視窗，解讀其背後潛在動機（例如：「*警示：業者 B00446 在 2026 年 3 月底頻繁將心臟節律器分流至 5 家關聯主體，行為符合典型『過期前趕進度洗貨』特徵，潛在囤積黑市風險機率為 87%。*」）。

### 5.3 AI Magic 3：一鍵生成 TFDA 標準稽查公文與報告書 (One-Click TFDA Audit Report Generator)

* **功能機制**：當使用者利用圖表與篩選器揪出一組漏報、違規的廠商與醫院名單後，點擊「一鍵產出執法公文」按鈕。
* **Wow 產出**：AI Agent 會調用內置的**台灣行政機關標準公文格式與語意範本**，自動帶入違規主體（如：台北榮總、臺灣百特）、精確違規事實（漏報哪些批號、漏報數量、申報延遲天數）、明確違反之法條（如：違反本辦法第 X 條第 Y 項，依醫療器材管理法第 Z 條處分之依據），直接在 Streamlit 介面上產出**字字珠璣、格式完美、可直接列印印發的稽查公文與綜合報告草卷**。

### 5.4 AI Magic 4：智慧欄位對齊與模糊勾稽引擎 (Smart Schema Alignment & Fuzzy Matching)

* **功能機制**：若使用者上傳的本地原始 CSV 欄位極度不規範（例如：欄位名稱被改為 `證書編號`、`產品型號_備註`、`單位(個/組)`），傳統系統會直接報錯。本系統利用 AI 的語意識別能力，在背景進行欄位對意圖（Schema Matching）分析。
* **Wow 產出**：AI 自動將異質欄位映射（Mapping）到標準 Schema 上，並進行模糊勾稽（如將中文品名 `美敦力亞士爾節律器` 與許可證主檔上的 `“美敦力” 亞士爾磁振造影植入式心臟節律器` 判定為 100% 同一產品），免去人工清洗的極大痛苦。

### 5.5 AI Magic 5：智慧型互動式數據偵探探針 (AI Data Detective Conversation)

* **功能機制**：一個常駐於 Streamlit 右側的對話面板。該 Agent 內置 **Code Interpreter（代碼解釋器）** 沙盒環境，並具備 RAG（檢索增強生成）能力，能即時讀取全量申報數據矩陣與法規手冊。
* **Wow 產出**：使用者可以直接輸入極度口語的中文 Prompt（如：「*幫我查出今年度所有節律器申報不一致的醫院名單，按漏報嚴重度排序，並順便幫我把這群醫院的流向圖畫出來*」）。Agent 會在背景**自動編寫並執行 Python 程式碼**，將計算出的精準表格呈現在對話框中，並直接動態生成對應的 Plotly 圖表，實現真正的「數據交談式稽查」。

## 6. Streamlit WOW UI 視覺與互動界面設計 (User Interface Design)

為顛覆一般政府系統或內部工具呆板、沉悶的印象，本平台 UI 採用深色現代科技調性（或高對比執法冷灰調），並嚴格遵循**網格系統與舒適排版美學**。

### 6.1 視覺主視覺色調與排版規範

* **主色調 (Primary & Neutral Colors)**：

## 7. 數據處理性能優化與安全隱私隔離

### 7.1 效能優化策略

* **記憶體內索引化（In-Memory Indexing）**：針對每年 10k+ 筆原始紀錄，系統導入時即以 `transaction_id` 與 `license_no` 建立複合索引（Multi-Index），確保在 Streamlit 執行重度篩選時，能在 `50ms` 內完成資料矩陣切片（Slicing）。
* **Polars 惰性求值（Lazy Evaluation）與多執行緒（Multi-Threading）**：當未來擴充至 100k+ 以上的百萬級全台累積大數據時，底層核心計算模組可無痛切換為 Polars 引擎，利用並行運算（Parallel Processing）徹底榨乾本地端 CPU 的多核心效能，防止網頁前端出現 Loading 卡死。

### 7.2 安全與隱私隔離

* **100% 本地運行（No Data Leaks）**：本技術規格嚴格禁止將任何原始醫材數據、醫院採購價格、序列號、或機構名冊上傳至任何第三方雲端資料庫。
* **LLM 隱私防護管道（Privacy-Preserving Prompting）**：在使用右側 AI 偵探對話或生成公文時，若涉及敏感資訊，系統內置的數據去識別化過濾器（De-identification Filter）會在將資料結構提交給本地模型（如 Ollama/Llama3）或合規遠端大模型之前，自動將實體名稱替換為虛擬代號（如：將「台大醫院」替換為「機構_A00013」），確保符合政府最高等級的資安合規標準。
