台灣醫器 UDI 與許可證合規分析系統：企業級技術規格與監管官員操作指南
本文件為 「台灣醫器 UDI 與許可證合規分析系統 (Taiwan Medical Device UDI & License Compliance Analysis System)」 的全方位技術規格書、架構設計藍圖、監管官員實務指南，以及未來前瞻 AI 監管特徵規劃。本系統旨在協助衛生福利部食品藥物管理署 (TFDA) 稽查官員、政策制定者與合規分析師，透過大數據分析與視覺化技術，精準掌握國內醫療器材單一識別系統 (UDI) 的實施現況、排除合規漏洞，並對未來許可證失效進行前瞻性預警。
1. 系統願景與監管背景
1.1 台灣醫療器材 UDI 監管背景
隨著《醫療器材管理法》的施行，台灣醫療器材單一識別系統 (Unique Device Identification, UDI) 已進入全面強制登載階段。依據法規，醫療器材許可證持有商必須在法定期限內，將醫療器材之基本單一識別碼 (Basic UDI-DI) 及相關產品資訊登載至指定的 UDI 資料庫中。
一級 (Class 1) 醫材：採逐步推進與自願申報。
二級 (Class 2) 醫材：絕大多數已被納入強制登載範疇，是市場監管的核心主力。
三級 (Class 3) 高風險醫材：屬於生命維持或植入性高風險器材，必須 100% 進行 UDI 登載，任何漏登均屬重大法規違規。
然而，在實務監管上，TFDA 官員面臨以下重大痛點：
數據孤島與串接困難：許可證資料庫與 UDI 註冊資料庫通常為兩個獨立的系統，官員難以即時判定某張許可證是否已履行 UDI 登載義務。
持有商規模與合規表現脫鉤：大型代理商與製造商旗下擁有數百張許可證，稽查人力無法手動評估每家企業的整體合規覆蓋率。
缺乏失效預警機制：許可證有其有效期限（通常為五年），當大量許可證即將到期，且其對應的 UDI 尚未更新時，將造成市場供應中斷或違規流通。
1.2 系統核心定位
本系統定位為 「情資驅動型監管決策儀表板 (Intelligence-Driven Regulatory Decision Dashboard)」。透過前端高效能特徵工程 (Feature Engineering) 與數據清洗，將雜亂的「許可證原始數據」與「UDI 登記數據」進行秒級對齊與關聯分析。系統不單純展示靜態報表，而是主動計算覆蓋率 (Coverage)、應登未登 (Critical Deficits)、失效預測 (Expiry Forecast) 等關鍵指標，為監管人員提供「一鍵式」的穿透化稽查路徑。

2.2.1 許可證字號標準化演算法 (standardizeLicenseNo)
由於歷史數據輸入習慣差異，許可證字號中常夾雜半形/全形空格、加號或大小寫變體。此演算法是跨資料庫「主鍵對齊」的關鍵：
code
TypeScript
export function standardizeLicenseNo(no: string): string {
  if (!no) return "";
  return no.trim().toUpperCase()
    .replace(/\s+/g, "") // 移除非預期半形空格
    .replace(/　/g, "")  // 移除非預期全形空格
    .replace(/[＋\+]/g, ""); // 移除前置或後置雜訊符號
}
2.2.2 智能日期解析引擎 (parseDateString)
許可證到期日來源多元，包含標準西元年（YYYY/MM/DD、YYYY-MM-DD）以及台灣特有的民國紀年（如 112/05/20、1120520）。

export function parseDateString(dateStr: string): Date | null {
  if (!dateStr) return null;
  const cleaned = dateStr.trim();
  
  // 模式一：標準西元年 YYYY/MM/DD 或 YYYY-MM-DD
  const matchObj = cleaned.match(/^(\d{4})[\/-](\d{1,2})[\/-](\d{1,2})$/);
  if (matchObj) {
    const year = parseInt(matchObj[1], 10);
    const month = parseInt(matchObj[2], 10) - 1;
    const day = parseInt(matchObj[3], 10);
    return new Date(year, month, day);
  }
  
  // 模式二：民國年格式 (如 112/05/20)
  const rocMatch = cleaned.match(/^(\d{3})[\/-](\d{1,2})[\/-](\d{1,2})$/);
  if (rocMatch) {
    const year = parseInt(rocMatch[1], 10) + 1911; // 轉換為西元年
    const month = parseInt(rocMatch[2], 10) - 1;
    const day = parseInt(rocMatch[3], 10);
    return new Date(year, month, day);
  }

  const standardDate = new Date(cleaned);
  return isNaN(standardDate.getTime()) ? null : standardDate;
}

許可證已過期：不論 UDI 登載狀況，只要有效日期小於當前系統時間（2026-07-12），即列為「許可證已過期」。此類許可證不納入當期合規覆蓋率的分母，避免數據失真。
完全合規：許可證在有效期內，且 basicDi 欄位存在有效登載值。
應登未登 (Critical Deficit)：許可證在有效期內，無任何 UDI 登載記錄，且該醫材級數為二級或三級。這屬於高風險違規，是稽查官員的首要約談與懲處對象。
資料漏登：許可證在有效期內，無 UDI 登載記錄，但該醫材級數為一級。由於一級醫材目前多屬輔助性質或自願登載，法規強制力較低，故將其定義為低風險之「資料漏登」，列為輔導改善對象。

4.2 監管實務應用案例分析
案例一：針對高風險（三級）重症醫材「應登未登」之精準查緝
政策目標：確保全台灣境內流通的心臟支架、人工關節等三級高風險醫療器材，100% 完成 UDI 登記，杜絕來源不明的灰色地帶醫材。
背景情境：
特派稽查官員「張組長」接獲密報，指出有部分高風險植入性醫材疑似未依法進行單一識別碼登載，可能在追溯鏈上產生漏洞。他需要立刻找出市場上所有不合規的三級醫材許可證與對應企業。
操作指南步驟：
進入系統首頁，視線落於頂部的 「Wow KPI 核心數據看板」。此時看板上的「合規登記覆蓋率」顯示全體合規率，但這是混雜了一級、二級的平均值。
移步至左側「篩選控制台 (Filters Panel)」：
在「醫療器材級數」中，取消勾選「1級」與「2級」，僅保留「3級」。

5. 三大前瞻性「Wow AI」智慧監管功能藍圖
為響應台灣「智慧政府 2.0」與科技監管政策，本系統規劃引入基於 Gemini 3.5 Pro/Flash 核心模型的 AI 監管特徵。這三大功能將徹底改變傳統由人工核對法規與合規報表的繁雜工作。
所有 AI 功能皆秉持 「API 金鑰安全保護」 與 「伺服器端中介代理 (Server-side Proxy)」 原則。API 金鑰安全儲存於伺服器環境變數 GEMINI_API_KEY 中，絕對不曝露於前端瀏覽器。
AI 功能一：智慧許可證合規自動稽核與監管建議報告生成助理
功能定義：
當官員在篩選條件下遇到複雜的合規缺口時，無需手動編寫公文與改善建議。AI 助理能自動掃描當前過濾後的 CleanedRecord[] 與 HolderStat[]，一鍵生成具備法律約束力語氣的「企業合規缺口診斷書」與「官方輔導公文草案」。
技術實現路徑：
前端將當前篩選出的合規異常數據（如某一高風險持有商的 32 張未登載證照明細）打包成 JSON。
透過安全 API 路由 /api/ai/audit-report 傳遞至 Express 後端。
後端調用 @google/genai SDK 的 gemini-2.5-pro 模型（適合處理複雜推理與法規文本撰寫）。
設定系統提示詞 (System Instructions)，融入台灣《醫療器材管理法》法條，指導 Gemini 以專業、嚴謹、不容妥協的法規稽查官員口吻撰寫報告。

// server/api/audit.ts
import { GoogleGenAI } from "@google/genai";
import express from "express";

const router = express.Router();

// 延遲初始化，避免無 Key 時系統崩潰，並確保金鑰隱蔽
let aiClient: GoogleGenAI | null = null;
function getAiClient() {
  if (!aiClient) {
    const apiKey = process.env.GEMINI_API_KEY;
    if (!apiKey) {
      throw new Error("系統環境變數中遺失 GEMINI_API_KEY，請於 AI Studio 設定中配置。");
    }
    aiClient = new GoogleGenAI({ apiKey });
  }
  return aiClient;
}

router.post("/generate-audit-report", async (req, res) => {
  try {
    const { holderInfo, unregisteredLicenses } = req.body;
    const ai = getAiClient();

    const prompt = `
      你現在是台灣衛生福利部食品藥物管理署 (TFDA) 的資深醫療器材法規稽查專家。
      請針對以下合規異常企業進行法規診斷，並撰寫一份官方限期改善輔導公文草案。

      [企業基本資料]
      企業名稱: ${holderInfo.name}
      統一編號: ${holderInfo.taxId}
      許可證總數: ${holderInfo.totalLicenses} 張
      未登載 UDI 許可證數: ${holderInfo.unregisteredLicenses} 張
      目前合規覆蓋率: ${holderInfo.coverageRate}%
      風險判定: ${holderInfo.complianceStatus}

      [違規許可證明細]
      ${JSON.stringify(unregisteredLicenses)}

      [法律與法規參考]
      根據《醫療器材管理法》第 25 條及相關 UDI 登載辦法，二、三級醫材未依規定登載單一識別碼者，得處以新台幣三萬元以上、五十萬元以下罰鍰，並得限期改善、按次處罰；情節嚴重者得廢止該許可證。

      [要求]
      1. 報告請使用繁體中文 (台灣繁體)。
      2. 語氣必須莊重、嚴謹、具備執法威嚴。
      3. 內容需包含：一、違規現況精準診斷；二、對應之《醫療器材管理法》法條適用性分析；三、具體改善步驟指引（包含 UDI 登載系統操作路徑）；四、罰則警告與限期改善截止日（設定為收文後 15 天內）。
    `;

    const response = await ai.models.generateContent({
      model: "gemini-2.5-pro",
      contents: prompt,
    });

    res.json({
      success: true,
      reportHtml: response.text,
    });
  } catch (error: any) {
    res.status(500).json({ success: false, error: error.message });
  }
});

6. 20 個深度研討與隨訪問題 (監管科技與系統演進)
為了推動本系統在監管科技 (RegTech) 領域的持續演進與未來系統規劃會議，以下提出 20 個涵蓋「法規、數據工程、系統架構、AI 整合、實務稽查」維度的深度研討問題，供 TFDA 技術委員會與政策分析小組進行研討：
數據工程與架構維度 (Data Engineering & Architecture)
問題一 (跨系統實時對齊)：當前系統依賴手動上傳許可證與 UDI 兩張 CSV 報表進行本地 Left Join。在未來的生產環境中，如何設計穩健的 Change Data Capture (CDC) 機制，實現 TFDA 許可證系統與全球 UDI 資料庫（如美國 GUDID、歐盟 EUDAMED）之秒級實時同步？
問題二 (大數據記憶體效能)：若台灣醫材許可證與 UDI 登記總量在未來達到數百萬筆（包含歷年歷史過期數據），前端 useMemo 與 PapaParse 的記憶體壓力將大幅上升。應如何在不破壞 client-side 純淨體驗的前提下，引入 IndexedDB 或 Web Workers 進行後台分流多執行緒運算？
問題三 ( ROC 與西元日期相容性)：目前 parseDateString 演算法雖然支援民國年與西元年解析，但部分極早期許可證存在民國「雙位數年（如 89/12/31）」或純文字「無限期」之狀況。如何建立更具容錯力的機器學習或正規表示式模型，確保 100% 日期解析成功率？
問題四 (資料清洗的防注入安全性)：當用戶上傳自訂的 CSV 檔案時，若檔案中夾雜惡意代碼（如 CSV Injection 攻擊，利用 =CMD() 觸發 Excel 漏洞）。系統如何在 PapaParse 階段實施嚴格的防沙箱過濾與欄位消毒（Sanitization）？
問題五 (一對多關係之物理展平限制)：目前若一張許可證對應多個 Basic UDI-DI，系統會將其展平成多筆 CleanedRecord。這在計算「許可證張數分母」時會產生重複計算的偏差。如何調整統計引擎，在「指標層面」維持唯一張數，而在「明細層面」維持完整 UDI 鏈結？
法規、監管與實務稽查維度 (Regulations & Inspection Practice)
問題六 (應登未登之法規灰色地帶)：部分二級醫療器材（如自願申報期間的一級醫材或法規豁免品項）在實務上並無強制登載 UDI 之義務。系統如何建立動態的「免登載白名單表」，避免稽查官員誤判並寄出不當的催告公文？
問題七 (製造商多國籍與地緣政治判定)：目前系統僅依據 製造廠國別（如 US, DE）進行單一長條圖分類，然而許多跨國大廠在多國設有分廠，或在中國大陸生產後轉由新加坡包裝。如何優化資料模型以支援「複合式產地與轉口鏈」之多維度視覺化分析？
問題八 (過期許可證之流通殘餘管制)：在實務中，許可證過期（如 2026/06 過期）並不代表市面上已無該產品，醫院可能仍有過期前進口之庫存。本系統應如何結合醫學中心之實際病患使用回報數據（UDI-PI 追溯），進一步交叉比對已過期證照的「實際流通殘餘風險」？
問題九 (發碼機構多樣性漏洞)：除了主流的 GS1、HIBCC 外，台灣部分中藥或特殊牙科醫材可能採用本土自行定義之識別碼。系統如何為這類「非主流/自定義發碼」設計合規評判基準，以防其被判定為「資料漏登」？
問題十 (持有商組織變革與統編漂移)：當兩家大型醫材公司合併時（例如公司 A 併購公司 B），其許可證轉移手續通常需要數月，期間會產生「統編仍為舊公司，但實際管理商已變更」的混亂期。系統如何記錄並呈現企業主體歷史變革的「時間軸足跡」？
前瞻 AI 與大語言模型整合維度 (GenAI & Agent Technologies)
問題十一 (AI 決策之可解釋性與法律免責)：當 AI 功能一 生成輔導改善公文時，若因語意理解偏差，誤植了罰款金額或引用了錯誤的法規條款，這在公文書信上是致命的。我們應如何設計 Human-in-the-loop (人工核決) 介面，讓稽查官員在公文發出前進行雙重確認？
問題十二 (Gemini API 脈絡長度與成本優化)：如果一次性將 10 萬筆醫材資料作為 Prompt 傳送給 Gemini 進行宏觀稽核，會消耗極高 Token 成本且容易超出脈絡長度（Context Window）限制。應如何採取 RAG (檢索增強生成) 或數據特徵預先壓縮（Embedding）技術，僅將最關鍵的「高危企業摘要」送入 AI 運算？
問題十三 (Google Search Grounding 的時效性校準)：在AI 功能二（供應鏈風險地緣引擎）中，Google Search Grounding 如何排除網路上的農場新聞或非官方謠言，確保引用來源皆為美國 FDA、歐洲 EMA、台灣 TFDA 等各國國家級藥物主管機關之官方公告？
問題十四 (私有數據隱私與資料不落地)：在處理涉及企業商業機密（如部分尚未公開之專利醫材 Basic UDI-DI 申請案）時，如何向企業與法規單位證明，送往 Google Gemini 伺服器進行語意分析的數據不會被作為未來大模型公開訓練的語料？
問題十五 (AI 多語系品名語意校對之局限性)：許多進口醫材中文名稱為音譯（如 "“席爾” 導尿管"、"“希爾” 導尿管"），且英文名稱常帶有大量複雜的醫學縮寫（如 PTCA, Balloon, Cath）。AI 語意對齊模組如何建立醫療器材專業術語字典，防範「幻覺 (Hallucination)」導致對齊錯誤？
系統推廣、生態系與未來擴展維度 (Ecosystem & Future Expansion)
問題十六 (大眾透明化與業者端開放 API)：本系統目前為 FDA 官員專用之監管後台。未來是否有可能擴展出一個「業者合規自助平台（業者端 View）」，允許醫材商上傳自家 CSV，在出貨前自行校驗 UDI 結構是否合規，將違規扼殺於進口關卡前？
問題十七 (與醫院端 UDI 掃描追溯系統之閉環對接)：全台各大醫院（如台大、榮總）在手術室使用醫材時皆會掃描 UDI 條碼。本系統如何與醫院端的「電子病歷與資材管理系統」對接，實現從「許可證發放 -> 業者進口登載 -> 醫院實體掃描 -> 病患體內植入」之 End-to-End 全生命週期閉環監管？
問題十八 (行動化稽查之響應式延伸)：當稽查官員實地前往進口商倉庫進行現場抽查（現場開箱比對條碼）時，目前的網頁版介面是否支援平版、手機等行動裝置之極致響應式操作？未來如何結合行動裝置相機，開發「直接掃描產品包裝 1D/2D 條碼，即時連回本系統驗證合規性」之延伸特徵？
問題十九 (國際 UDI 法規變更之自適應彈性)：全球 UDI 規範不斷演進（例如 GS1 正在逐步推廣新一代的 2D DataMatrix 代替傳統一維條碼）。本系統在 UI 與資料模型設計上，保留了多少彈性以相容未來發碼格式的劇烈變動？
問題二十 (歷史合規演進之時間維度軌跡 (Time-traveling))：目前系統為「當下狀態分析」（以 2026/07/12 為當前基準）。如何建立「合規歷史軌跡資料庫」，讓官員能查看「這家企業在過去三年內的合規覆蓋率走勢」，進而判定該企業是「持續改善」還是「常態性擺爛」之違規慣犯？
7. 總結 (Executive Summary)
本 「台灣醫器 UDI 與許可證合規分析系統」 技術規格與指南，展示了如何將現代 Web 前端技術（React 19、Vite、Tailwind CSS、Recharts、Motion）與精準的監管數據工程（特徵工程、Left Join 演算法、企業主體 Roll-up）完美結合。
本系統摒棄了低效、冗長的傳統人工作業模式，賦予 TFDA 官員以下核心監管武力：
一秒穿透：從宏觀的全國 UDI 覆蓋率，一鍵切換至特定違規企業及其旗下高危三級許可證清單。
前瞻預警：透過 24 個月時間序列预测折線圖，提早 60 天鎖定即將失效之關鍵醫療器材。
AI 賦能：規劃引進 Gemini 3.5 智慧引擎，自動化撰寫稽查公文，並即時串接國際地緣政治與安全回收情報。
本系統的建置與推廣，將協助台灣醫療器材監管邁入「情資驅動、精準治理、科學決策」的全新智慧法規時代，守護全體國民的用藥與醫療器材安全。
