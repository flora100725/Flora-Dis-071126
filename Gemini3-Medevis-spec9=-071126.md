第一部分：系統技術規格書 (Technical Specification)
1. 系統願景與設計哲學 (System Vision & Design Philosophy)
1.1 研發背景
在全球化醫療器械供應鏈中，監管機構面臨著前所未有的挑戰：
標準不一：台灣本地的醫療器材單一識別系統 (UDI) 與登錄資料庫 (TUDID) 在分類、術語及管理維度上，與世界衛生組織 (WHO) 推行的全球醫療設備資訊系統 (MeDevIS) 存在術語鴻溝。
資訊孤島：醫院採購端、代理商申報端與衛生主管機關監管端的資訊未能即時互通，導致異常採購、不合規器材流通及不良反應事件（Adverse Events）的響應速度受限。
稽核成本高昂：傳統審查高度依賴人工比對海量申報清單與仿單，效率低下且難以識別深層次的關聯風險。
1.2 系統核心定位
TUDID-MeDevIS Agent OS 是全台首創、對標國際的雙軌主動式醫療器材智慧代理人監管系統。系統將台灣 TFDA 的 TUDID 申報數據與 WHO MeDevIS 資料庫深度融合，透過代理人運算圖譜 (Agentic Workflow)、即時網路 Grounding 檢索與語意化網路分析 (Semantic Graph Analytics)，建構一個兼具臨床合規性、採購決策支援及全球安全警戒的次世代監管平台。
1.3 設計美學與人因工程
本系統拋棄了傳統公務機關資訊系統繁雜、生硬的表格堆砌，採用 Space Grotesk 科技無襯線字體與 JetBrains Mono 等寬數據字體進行排版。界面配色基於高對比度的深邃星際灰（Cosmic Slate, #09090b 至 #27272a），輔以符合人因工程的動態半透明磨砂玻璃質感（Glassmorphism），使監管官員在長時間面對海量申報數據與關聯圖譜時，仍能保持極高的專注度，降低視覺疲勞。

3.1 前端展示層 (Client-Side SPA)
技術棧：React 19, TypeScript, Tailwind CSS v4, Motion (動態庫), Recharts & D3.js (數據視覺化)。
核心功能：
無縫單頁面佈局 (Single-View Layout)：所有監管功能（儀表板、數據匯入、沙盒測試、AI 協同代理人、Wow 核心）均在一體化界面中響應式切換，避免頻繁刷新頁面導致監管官員上下文遺失。
D3.js 關聯拓撲圖：直觀渲染進口商、醫材、醫院之間的物流與金流關聯。
零洩密安全隔離：前端不儲存、不調用任何 API 金鑰，所有對 AI 模型或外部資料庫的調用均經由後端代理。
3.2 後端業務邏輯層 (Express Server-Side Backend)
技術棧：Node.js, Express, tsx (TypeScript 執行引擎), @google/genai SDK。
核心功能：
Gemini 3.5 雙向整合代理：在 server.ts 中，使用最新的 @google/genai 軟體開發套件。透過延遲加載（Lazy Initialization）與嚴格的環境變數稽核，在服務端安全調用 gemini-3.5-flash 模型。
即時網絡 Grounding (搜尋落地)：配置並啟用 googleSearch 工具，允許 AI 代理人在回答監管官員提問時，即時檢索 TFDA 官網、WHO 官網、FDA (美國食品藥物管理局) 的最新醫療器材召回與合規公告。
標準化數據處理管道 (Data Ingestion Pipeline)：解析上傳的 JSON/CSV 資料，自動識別缺失欄位，利用規則引擎與大語言模型對數據進行雙軌制清洗（Data Sanitization）。
3.3 數據儲存與安全邊界
地端暫存與雲端持久化：
採用 localStorage 保存用戶非敏感的個性化界面配置、會話臨時日誌與本地技能測試代碼。
生產環境可無縫對接 Google Firebase Firestore（提供即時多用戶協同監管與安全稽核軌跡）或 Cloud SQL (PostgreSQL)（處理複雜的結構化採購防弊關聯查詢）。
4. 系統核心組件技術解析 (Core Components Analysis)
4.1 數據匯入與清洗標準化組件 (DataIngestion.tsx)
這是系統的「數據入口處」。當 FDA 官員或申報廠商導入混雜的申報單、採購單或物流清單時，該組件會：
結構解析：支持 Drag-and-Drop 拖放，利用前端 File API 進行數據流式讀取。
AI 主動診斷：將前 10 筆樣本數據傳送至後端 Express API，由 Gemini 模型分析數據結構，預測各欄位對應的 TUDID / MeDevIS 標準實體，並給出「置信度評分（Confidence Score）」。
錯誤欄位高亮：若偵測到格式錯誤（如日期不合規、許可證字號字數不對、缺漏 UDI 碼），以高對比度的警示色（如深橘 #f97316）高亮標示，供官員快速手動修正。
4.2 智慧節點圖譜與決策儀表板 (GraphDashboard.tsx)
為了將枯燥的二維表格轉化為直觀的監管情報，該組件採用了 D3.js 力導向圖（Force-Directed Graph） 與 Recharts 統計圖表：
節點類型：
藍色節點 (Supplier Node)：代表進口商或本地製造商。
黃色節點 (Device Node)：代表醫療器材實體（以 UDI-DI / 許可證字號唯一識別）。
綠色節點 (Hospital Node)：代表全台各級醫療機構。
邊的語意 (Edge Semantics)：
邊的粗細代表交易金額或配送數量。
邊的顏色代表合規狀態（綠色代表正常，紅色代表該批次醫材存在國際召回警戒）。
交互過濾器 (Cross-Filtering)：點擊任一節點，整個儀表板（包括漏斗圖、長條圖、趨勢圖）將即時連動，過濾出與該實體相關的所有歷史紀錄。
4.3 協同 AI 代理人副駕駛 (ChatInterface.tsx)
這是一個基於大語言模型（LLM）的智慧問答終端，它不僅僅是一個聊天視窗，而是一個監管決策輔助系統：
推理步驟可視化 (Chain-of-Thought logs)：當官員輸入「查詢這批人工關節是否有國外回收紀錄」時，視窗上方會流式（Streaming）輸出代理人的決策圖譜日誌（例如：[解析許可證] 衛部醫器輸字第012345號 -> [檢索 WHO MeDevIS] 對齊國際 UDI-DI... -> [執行 Google Search] 搜索 FDA 回收公告...）。這極大地增加了 AI 回答的可信度與透明度。
結構化資訊卡片 (Structured Entity Cards)：對於對齊成功的醫材，AI 會在對話中直接渲染一張富文本卡片，包含其 WHO 分類、本地許可證狀態、全球平均採購單價區間等，並附帶一鍵導出為稽核報告的功能。
4.4 技能沙盒測試與 PlayGround 實驗室 (Playground.tsx)
本組件為技術官員提供了一個無需部署即可測試自定義監管規則的「代碼沙盒」：
動態代碼編譯：利用 React 狀態引擎，技術官員可以直接在界面中編寫 TypeScript 自定義過濾規則或 AI System Prompt，並即時應用於當前加載的醫材資料庫中，觀察篩選結果。
規則可視化對比：支持雙視窗對比，一邊是標準 TFDA 邏輯，另一邊是結合了 WHO MeDevIS 標準的雙軌制邏輯，直觀展示雙軌制如何降低「漏判率」與「誤判率」。
5. 資訊安全、存取控制與隱私保護 (Security, ACL, and Privacy)
作為醫療級監管平台，數據隱私與合規性是系統命脈：
數據傳輸加密 (Transit Security)：所有客戶端與伺服器之間的通訊，強制實施 TLS 1.3 級別加密。
零密鑰外洩 (Zero-Secret Leakage)：
系統敏感的金鑰（如 GEMINI_API_KEY）絕對不寫死在代碼中，而是統一配置在環境變數 .env 中，由 AI Studio 容器或生產環境主機安全注入。
前端 React 代碼絕不引用 process.env.GEMINI_API_KEY，防止在瀏覽器開發者工具（DevTools）中外洩。所有 AI 請求均發送至後端的 /api/chat，由後端向 Google AI 伺服器發起請求。
基於角色的存取控制 (Role-Based Access Control, RBAC)：
系統在 LoginModal 中定義了三種核心監管角色：
FDA_OFFICER（食藥署稽核官員）：擁有最高權限，可讀取全台採購流向、不良反應警示、並調用 Playground 調整 AI 稽核權重。
HOSPITAL_ADMIN（醫療機構管理員）：僅能查看與上傳該院所內部的採購與分發紀錄，並使用 AI 副駕駛優化內部合規性。
SUPPLIER_REPRESENTATIVE（申報業者代表）：僅能申報其代理的醫材數據，查看 TUDID 與 WHO MeDevIS 的對齊進度，無權查看其他競爭廠商的銷售價格或分發院所明細。
病患隱私去識別化 (PII Sanitization)：
在數據匯入（Ingestion）階段，系統內建 PII（個人識別資訊）過濾機制。若上傳的原始分發紀錄中含有病患姓名、身分證字號或病歷號，系統會自動在前端使用 SHA-256 算法進行不可逆雜湊（Hashing），或直接遮蔽（Masking），確保系統數據庫中僅保留「匿名化」的臨床分發統計。

2.1 綜合監管案例 1：醫療器材供應鏈溯源與許可證合規性審查
2.1.1 監管背景與痛點
某家大型進口代理商（供應商代碼：SUPPLIER_01）向食藥署申報了一批高風險的「心臟支架（Cardiac Stent）」。然而，傳統的紙本或關聯式資料庫申報格式不一，且稽核官員難以判定該型號是否在全台各家醫院存在「價格哄抬」或「許可證過期」之情事。
2.1.2 系統操作流程

Q7：在 Playground 技能測試中，如何防止技術官員編寫出「邏輯無限循環」或「耗盡 API Token 額度」的惡意或錯誤代碼？
深度分析：Playground 允許官員自定義 TypeScript 規則與 System Prompt。這帶來了極大的靈活性，但也帶來了系統穩定性挑戰。如果編寫的代碼包含未終止的遞迴、或是向後端 API 發起海量的並行請求，可能會導致服務端當機（DoS）。系統應如何限制沙盒的「運算時間配額（Execution Timeout Limit）」、「記憶體上限（Memory Cap）」及「API 呼叫速率限制（Rate Limiting）」？

深度分析：當 FDA 稽核官員否定了 AI 代理人的一項判定（例如：AI 判定某超音波儀價格異常，但官員查證後發現其包含特殊的跨國保固，判定為合規），系統應如何安全地記錄這次「人類專家的糾正反饋」？在不重新訓練大模型的前提下，如何透過「動態 RAG（檢索增強生成）知識庫更新」或「上下文少樣本學習（Few-Shot Context Ingestion）」，讓 AI 代理人「越用越聰明」，自動適應台灣本地的最新監管心智模型？
3. 大規模並行處理與數據工程 (Large-Scale Processing & Data Engineering)
Q9：當全台醫院每日申報與配送數據量達到數十萬筆時，系統目前的記憶體暫存（In-Memory）架構應如何擴展，以確保 D3 拓撲圖與 Recharts 仍能維持 60 FPS 的流暢渲染？

深度分析：許多中小型醫院保存的舊合約與發票仍是手寫或蓋章覆蓋的掃描檔。傳統 OCR 的識別率極低。我們應如何調配 Gemini Pro 視覺模型的「圖像超解析度（Super-Resolution Pre-processing）」與「上下文語意糾錯（Contextual Spelling Correction）」技術，使其即使在文字缺失 30% 的極端狀況下，仍能精準推測出正確的醫療器材規格型號？

4. 資訊安全、極致防洩密與零信任 (Security, Anti-Leakage & Zero-Trust)
Q13：為了通過台灣最高等級的「政府資通安全防護基準 (A級/B級)」評測，本系統在前端、後端及傳輸端需進行哪些加固措施？

Q14：當 FDA 官員攜帶行動裝置（如 iPad）進入醫院進行現場稽查時，系統如何防止「螢幕截圖外洩（Screenshot Leakage）」與「旁路物理拍照外洩」？
