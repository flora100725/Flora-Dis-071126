TFDA Agentic OS 醫材流向與追溯智能治理終端：完整技術規格、稽查官使用指南、WOW AI 功能提案與隨訪問題
本文件為 Agentic OS 醫療器材流向與追溯智能治理終端（TFDA Agentic OS Distribution Hub） 的技術規格書與操作指南。本系統專為中華民國衛生福利部食品藥物管理署（TFDA）設計，用於監控高風險醫療器材（如心臟節律器、人工關節及藥物洗脫支架）之全國流向，確保其符合《醫療器材來源流向資料建立及管理辦法》之規定。
第一部分：系統技術規格書 (System Technical Specification)
1. 產品願景與設計美學 (Product Vision & Design Aesthetics)
1.1 產品定位與合規願景
本系統為政府監管級別之自動化法規審計與數據流向追蹤平台。其核心價值在於將傳統被動、零散的書面申報模式，升級為主動式、代理人驅動（Agent-driven）的智能監管終端。系統整合了 Google Agent Development Kit (ADK) 與 Gemini 尖端大語言模型，透過「智能代理人 (Agent)」自主分析、程式碼沙盒執行及網頁即時搜尋，大幅縮短異常流向調查時間，將合規精準度指標提升至 94% 以上，確保高風險植入式醫材在台灣境內「來源可溯、流向可追、風險可控」。
1.2 設計美學與界面布局
系統界面基於 Vanguard Grid 佈局架構，排除一切不必要的模擬終端裝飾或低效 AI 雜訊（Anti-AI-Slop），提供極致純粹且資訊稠密的專業級界面：
色彩與對比 (Contrast & Palette)：系統預設採用高對比度的深色「宇宙板岩 (Cosmic Slate)」主題（以 #0B0C10 深黑為基底，搭配 #00F5A0 霓虹薄荷綠作為核心高亮提示色，以及 #1E272C 作為卡片背景）。同時支援「流暢光學白」等 10 種團隊專屬風格 Preset，邊框與排版線條均採極簡 1px 細線（Border Slate-800/80），確保長時間觀看之眼部舒適度與專業度。
字型美學 (Typography)：標題選用極富未來科技感的 Space Grotesk，數據展示與終端日誌控制台則採用 JetBrains Mono 等寬字型，主文字則使用通用高辨識度之 Inter。透過字重（font-weight）與字距（letter-spacing）的精準微調，建立明確的資訊層級結構。
介面適應性 (Responsive Design)：支援桌面優先之高密度看板展示，在大螢幕下自動擴展為多欄式控制中心；在移動端則自動切換為流暢的單欄滾動模式，並維持 44px 以上的觸控目標大小。

2.1 Google ADK 工具箱深度整合技術细節
系統利用 @google/genai TypeScript SDK 的原生架構，將 Agent 提升為具有主動執行能力的「系統調查員」，突破傳統 LLM 僅能處理靜態知識的局限。
2.1.1 谷歌即時搜尋工具 (google_search)
作用機制：當 Agent 在稽查醫材許可證（如「衛部醫器輸字第030747號」）或調查國外製造商（如 Medtronic、Baxter）時，若發現資料庫中缺乏最新召回（Recall）資訊，或需要查核最新的 TFDA 公告，系統會自主決定調用 google_search 工具。
技術實現參數：
code
TypeScript
import { GoogleGenAI } from "@google/genai";
const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY });

// 伺服器端調用：啟用搜尋 Grounding
const response = await ai.models.generateContent({
  model: "gemini-3.1-flash-lite",
  contents: "查核醫療器材許可證『衛部醫器輸字第030747號』之最新主動召回或安全警訊狀態。",
  config: {
    tools: [{ googleSearch: {} }] // 注入搜尋工具
  }
});
視覺化對齊 (Source Grounding Visualizer)：前端接收到帶有 groundingMetadata 的響應後，會在 Agent 的文字輸出下方動態渲染「實時資訊源對齊卡片」，詳細列出網頁標題、發布時間、來源 URL（如 TFDA 官網、美國 FDA MAUDE 資料庫）及語意相似度，供稽查官一鍵點擊核實。
2.1.2 內置代碼執行沙盒 (built_in_code_execution)
作用機制：當稽查官導入數萬筆申報進出貨明細（CSV / JSON）並要求 Agent 「找出有批號異常或庫存不平衡的醫療機構」時，Agent 將不會使用不穩定的語意猜測，而是自動編寫出邏輯嚴密的 JS / Python 程式碼，並在雲端安全隔離的沙盒環境中運行。
技術特點：沙盒具有極高的安全限制，防止代碼對主機系統進行越權存取。運行結果（stdout/stderr 以及生成的文件、圖表）會由 Express 後端捕獲並結構化封裝，回傳給前端渲染。
3. 核心模組一：Agentic OS 智能治理終端與 Skill 生命週期 (Core Module 1)
3.1 skill.md 定義與解析引擎
在 Agentic OS 中，**「Skill (技能)」**是驅動 Agent 行為的靈魂。它以標準 Markdown 結構編寫，明確界定了 Agent 的角色定位、法律約束、輸入格式、推理邏輯、工具調用準則及預期的結構化輸出。
系統內置之三大核心 Skill 範本：
TFDA 法規合規稽查官 (TFDA Compliance Auditor Skill)：專門負責掃描進出貨賬目，查核許可證有效期限，對照法規條文。
Class III 植入物序號追溯專家 (Implant Serial-Number Traceability Expert Skill)：專精於心臟節律器、人工關節等高風險 Class III 醫材的唯一產品序號（Unique Serial Number）與批號鏈條查驗，防範竄改與伪冒。
主動式供應鏈失衡預測員 (Proactive Supply-Chain Forecasting Agent Skill)：利用程式碼執行沙盒，對各大醫院的消耗速率與分銷商庫存進行預估，預警缺貨風險。
3.2 技能生命週期管理功能 (Lifecycle Management)
多渠道導入：稽查官可直接上傳 .md 文件，亦可利用界面內建的高級 Markdown 編輯器進行實時編寫。編輯器支援語法高亮、行號顯示與自動排版。
LLM 輔助優化 (Agent-Assisted Skill Refinement)：稽查官可以使用自然語言命令 Agent 修改 Skill。例如輸入：「請幫我修改這個 Skill 的判定邏輯，當許可證效期剩餘小於 180 天時，即自動將狀態標記為黃色警告，而非原本的 90 天。」系統會調用 Gemini 生成修改後的 Markdown 文本，並提供 Git-Diff 雙欄視角，直觀對比修改前後的行位差異，經確認後一鍵覆寫。
一鍵導出：支援將當前調試完畢的 Skill 導出為 .md 文件，或以標準 JSON 格式下發至其他審計終端。
3.3 認知狀態步進器、實時日誌終端與指標面板
為消除 AI 決策的「黑箱效應」，工作區配備了高透明度的觀測工具：
** stethoscopic 步進狀態圖 (Cognitive State Stepper)**：採用節點圖（Node Map）形式，以呼吸燈微動畫即時動態展示 Agent 當前的思考階段：
[空閒 (Idle)] ➔ [解析申報帳目] ➔ [調用谷歌搜尋許可證] ➔ [啟動沙盒驗算 SN 唯一性] ➔ [生成法規合規意見書] ➔ [完成].
實時 ANSI 日誌終端 (ANSI Live Console)：客製化構建的虛擬終端，以不同顏色對應 Agent 的大腦活動（藍色代表 Inner Monologue 內省、紫色代表 Google Search 搜尋中、綠色代表 Code Execution 成功執行、紅色代表法規不合規警報）。終端支援一鍵清空、全文檢索及日誌層級過濾（Debug / Info / Warning / Error）。
認知指標儀表板 (Cognitive Speedometers)：三個高亮渲染的半圓形儀表，即時顯示：
思維輸出速率 (Thinking Velocity)：以每秒輸出 Token 數（t/s）展示。
工具調用延時 (ADK Tool Latency)：顯示搜尋與代碼執行所耗費的時間（ms）。
依據對齊率 (Source Grounding Score)：代表 Agent 輸出的論點中，被權威法規源明確 Grounded 的百分比。

4.2 雙向數據標準化清洗管道 (Standardization Pipeline)
在實際執法中，申報業者上傳的數據往往格式混亂（例如日期寫成 4/29/26、許可證號夾雜空格等）。本系統設計了高智能的清洗管道：
特徵掃描 (Signature Scanner)：當上傳非標準格式之 CSV、Excel 或 JSON 時，系統自動比對表頭特徵。
LLM 語意映射：對於無法匹配的自訂表頭（例如將「產品序號」寫成 SN_Code 或 序列號），系統在後台調用 gemini-3.1-flash-lite 進行語意對齊，自動轉換為標準 API 屬性。
沙盒強類型轉換 (Type-Casting Sandbox)：
將多種日期格式統一轉換為 YYYYMMDD 的整數或 ISO 字串。
自動校驗 UDI Barcode 的校驗碼（Checksum）。
衝突對比 UI (Mapping Diff View)：前端渲染「標準化衝突對比面板」，黃色標示被 LLM 推測轉換的表頭、紅色標示無效數據（如負數數量或不合規的許可證號），稽查官可手動覆核調整。
數據網格編輯器 (Interactive Grid)：標準化後的數據載入高達 500 行的實時表格，支援行內編輯（Inline Edit）、整列刪除、手動插入新申報項及一鍵導出標準 CSV。
4.3 專業級執法過濾矩陣 (Regulatory Filters)
為協助稽查官在汪洋般的交易海中精確落鎖，系統提供了由六大維度交織的篩選面板：
時間區間滑塊 (Temporal Slider)：支持精確至單日的雙向時間軸滑動，並提供「Q1 申報季」、「2026 年度累計」、「效期倒數 90 天」等快速篩選預設。
許可證號自動補全 (Autocomplete License Selector)：與許可證 Master 數據實時連動，隨打即搜，快速鎖定特定許可證的所有交易。
型號與批序號精準過濾 (Model & SN Wildcards)：支援正則表達式與模糊查詢，用於追蹤特定的高風險產品序號（Unique SN）。
申報主體與上下游節點追索 (Declaring & Trade Entity Tracker)：一鍵篩選出特定的進口商、分銷商或最終收貨醫院（如「國立台灣大學醫學院附設醫院 A00013」）的所有進出貨閉環。
醫材風險級別 (Risk Level Category)：直接隔離 Class II（中風險）或 Class III（高風險植入物）明細。

問題 07：Class III 高風險醫材若在手術室被緊急拆封使用，但事後醫院漏報了唯一序號（Unique SN）申報，本系統的「預測與主動重分配引擎」如何動態識別出這種「隱性消耗」，並在預測模型中進行自我校正？

問題 10：Sankey 桑基圖在展示「多級經銷」時，如果存在「循環流向」（例如分銷商 A 退貨給分銷商 B，B 又轉售給 A），會導致標準 Sankey 算法崩潰。我們應該如何調整拓撲排序算法，以支援這種雙向或循環的醫材流動可視化？

問題 12：如何在 GIS 地圖中無縫接入即時路況或天氣 API，以預測當颱風等天災來襲時，高風險急救醫材在實施跨地區緊急調撥時的實際物流延遲與最優運輸路徑？

問題 16：在自動生成處分公文與回收令時，AI 如何確保其引用的法規條文與最新頒布的台灣法規庫（MOHW Regulations Database）保持絕對一致，避免出現 LLM 常見的「法規條文幻覺（Hallucination）」？
